"But you will receive power when the Holy Spirit has come upon you." — Acts 1:8

Purpose

Acts contains all the frontend JavaScript that powers the FlockOS user interface. These files are loaded by the HTML applications (in Revelation) and handle everything the user sees and interacts with: API communication, module rendering, visual themes, authentication, and specialized tools like the worship music stand.

Everything is vanilla JavaScript — no React, no Vue, no npm. A lightweight Python minifier (rjsmin) produces the .min.js production files, but no bundler or framework toolchain is required.

Key Architecture: Module System

The module system is the core pattern of the frontend. Each module is registered with:

_def('moduleName', async function(el) { ... })

When a user navigates to a module, the framework calls this function, passing the DOM container element. The function fetches data from the API, builds HTML, and inserts it into the container. No routing library needed — just a view name and a render function.

Module Helper Functions

the_tabernacle.js provides a toolkit of shared helpers that every module uses:

  • _shell(el, title, subtitle, actions) — Creates the standard module header
  • _table(headers, rows) — Generates a styled data table
  • _body(el, html) — Sets the module body content
  • _badge(text, type) — Creates colored status badges
  • _edit(id, fields, updateFn) — Opens an edit modal with form fields
  • _del(id, deleteFn) — Confirmation dialog for record deletion
  • _rows(response) — Normalizes API response to array

Conventions

  • No external dependencies — everything is self-contained
  • All API calls go through TheVine — never call fetch() directly
  • Module IDs are kebab-case — e.g., prayer-admin, my-flock
  • _edit() always sends params.id — the backend must accept ID-based lookups
  • CSS classes use module prefixes — e.g., .ms- for MusicStand, .sys-cfg- for config

Implementation Notes

These JavaScript files are embedded directly into the HTML applications via <script defer> tags. In the Google Apps Script deployment model, they can be served as HTML includes or loaded from an external host.

Minification & Build

Every JavaScript file in Acts is maintained in two forms:

  • Source (.js) — Human-readable code with clear variable names, section comments, and logical whitespace. This is the file you edit and troubleshoot.
  • Minified (.min.js) — Whitespace, comments, and redundant syntax stripped out. This is what the HTML pages load in production.

The minification step delivers an estimated 30% reduction in file size (the full Acts bundle drops from ~1.57 MB to ~1.10 MB), which translates to roughly 30% faster page loads. Smaller payloads also produce more consistent load times across platforms — desktop Chrome, mobile Safari, Android WebView, and other browsers all benefit from reduced parsing and transfer overhead.

Build Tool

From the FlockOS project root, run:

./minify.sh

This activates the Python virtual environment and runs minify.py, which uses rjsmin to regenerate every .min.js file alongside its source. Run it after any code change.

Development vs. Production

During development, you work exclusively with the readable source files. The code is deliberately written to be easy to trace and troubleshoot — clear names, section headers, and descriptive comments. When ready to deploy, run the minifier; the HTML pages already reference .min.js, so no other changes are needed. The source files remain in the repository as the single source of truth.
