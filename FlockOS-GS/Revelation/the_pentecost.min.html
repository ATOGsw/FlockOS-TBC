Authentication & Session Flow

All auth is handled by two frontend modules (firm_foundation.js + the_true_vine.js) talking to the Flock CRM API.

User visits the_wall.html
  → firm_foundation.js: Nehemiah.guard()
      → sessionStorage has valid token?  YES → redirect to index.html
                                             NO  → show login form

User submits email + passcode
  → firm_foundation.js: Nehemiah.login(email, passcode)
    → the_true_vine.js: TheVine.auth.login(email, passcode)
      → POST to FLOCK_URL?action=auth.login
        → Auth.gs: login()
          → Hash.gs: hashPasscode(passcode, salt, pepper) — SHA-256
          → Compare hash to AuthUsers tab
          → Check AccessControl: role + status='approved'
          → Return {ok:true, session:{token,email,role,expires}, profile:{...}}
      ← sessionStorage.setItem(SESSION_KEY, JSON.stringify(session))
      ← sessionStorage.setItem(PROFILE_KEY, JSON.stringify(profile))
  ← redirect → index.html

Every API call (after login)
  → the_true_vine.js auto-appends token + email to each request
    → Auth.gs: validateToken() — checks AuthUsers.token + expiry
    → requireRole(e, 'leader') gates each handler by RBAC level

Session expiry (default: 6 hours)
  → TheVine returns {ok:false, code:'AUTH_EXPIRED'}
  → firm_foundation.js clears sessionStorage → redirects to the_wall.html

PROFILE_KEYthe_true_vine.js _configflock_auth_profileDisplay name, email, etc.

10 Troubleshooting

The #1 Rule

🚨 If something isn't working, count your files. Each project should have:

Matthew = 1  |  Mark = 1  |  Luke = 1  |  John = 1

Every project gets exactly one file (Code.gs), containing the respective Combined .gs. If you have more files than that, delete the extras.

"Unknown action"

The action parameter doesn't match any route. Check spelling — actions are case-sensitive and use dot notation (e.g. members.list, not membersList).

ReferenceError: [function] is not defined

All 24 John modules now live in one file — John_Combined.gs. Any file can call any function in the same global scope. This error means a function name was mistyped or renamed. Known bugs already fixed:

  • newId is not defined → Fixed: seedTheologyCategories_ in Theology.gs now calls generateId() (defined in John.gs).
  • ensureTab is not defined / addDropdowns is not defined → Fixed: both functions added to Utilities.gs (were previously only in the other three API projects).
  • resourceRowToObject is not defined → Fixed: renamed to compassionResourceToObj_() in Compassion.gs and discipleshipResourceToObj_() in Discipleship.gs to avoid collision.
  • isModuleEnabled is not defined (in Expansions context) → Fixed: renamed to isExpansionModuleEnabled_() in Expansions.gs; Auth.gs keeps the primary isModuleEnabled().

If a new ReferenceError appears, search all 24 files for the function name — a duplicate at the top level will silently shadow the one you expect.

"SyntaxError: Illegal duplicate" / Duplicate function name

Two files in the same GAS project define the same function name. GAS allows the project to save but throws at runtime. Fix: rename one of the duplicates (suffix with the module name and a trailing underscore, e.g. compassionResourceToObj_). Known duplicates already resolved — see above.

CORS / Cross-Origin Errors

GAS handles CORS automatically when "Who has access" = Anyone. If you see CORS errors, verify it's not set to "Anyone with Google account."

"Access denied" / 403

Session token is invalid/expired, or your RBAC role is too low. Re-login or check AccessControl tab.

Session Expired / Login Loop

Sessions expire after 6 hours (configurable via AppConfig SESSION_TTL_HOURS). Clear sessionStorage and re-login.

setupExpansion() Fails

Most common cause: SHEET_ID not set or wrong. Go to ⚙ Project Settings → Script Properties and verify it matches your Google Sheet URL.

Pepper Lost

If the pepper is lost, ALL passwords are unrecoverable. You must: clear all hashes, have every user set a new passcode, then run migrateAllPasswords() again.

11 Maintenance & Operations

"Be diligent to present yourself approved to God, a worker who has no need to be ashamed." — 2 Timothy 2:15

Add a New User

  1. Open the Flock CRM Google Sheet → go to the AuthUsers tab
  2. Add a new row with the user's email (col A), a temporary plain-text passcode (col B), role (col G = volunteer/leader/pastor/admin), status (col H = active), and timestamps (cols I–J)
  3. Add a matching row to the AccessControl tab (email, role, name, group, status='approved')
  4. In the GAS editor (Flock CRM API), open Hash → run migrateAllPasswords → confirm log shows "Migrated: [email]"
  5. Give the user their email + passcode → they can log in at the_wall.html

Change a User's Role

  1. Open Flock CRM → AccessControl tab
  2. Find the user's row → update col B (role) to the new value
  3. The change takes effect on the user's next login (or when their current session expires)
⚠ Also update col G in AuthUsers to keep both tabs in sync.

Disable / Suspend a User

  1. In AuthUsers: set col H (status) to suspended
  2. In AccessControl: set col E (status) to suspended
  3. In AuthUsers: clear col C (token) to immediately invalidate any active session

Update Deployed Code (After a Bug Fix)

  1. Edit the relevant .gs file in VS Code (FlockOS-GS/[folder]/)
  2. Open the corresponding GAS project at script.google.com
  3. Select the affected file in the left sidebar → select all → paste the updated contents
  4. Click Deploy → Manage deployments → ✏️ pencil icon
  5. Change "Version" to New version → Deploy
  6. URL stays the same — no config changes needed in TheVine

Add a New Module to John

If you add a new module (e.g. Events) to the Flock CRM API:

  1. Add the new functions to Database/John_Combined.gs in VS Code
  2. In the GAS editor, select all contents of Code.gs and replace with the updated John_Combined.gs
  3. Add setup logic to the Setup section if the module needs new spreadsheet tabs
  4. Add routing to the Api section so the action dispatcher knows about the new handlers
  5. If the module has helper functions that could collide with another module (e.g. resourceRowToObject), suffix them with the module name and a trailing underscore: eventsResourceToObj_
  6. Deploy a new version (see above)

Enable / Disable an Expansion Module

Module flags live in the AppConfig tab of the Flock CRM sheet. To toggle a module:

  1. Open Flock CRM → AppConfig tab
  2. Find the row for the module (e.g. MODULE_MISSIONS)
  3. Set value col to true or false
  4. Changes take effect immediately on the next API call (no redeploy needed)

Full Redeploy from Scratch

Use this checklist when rebuilding on a new Google account or if a project is corrupted:

  • Create 4 new Google Sheets (Flock Content, Flock Missions, Flock Statistics, Flock CRM)
  • Create 4 new GAS projects following Sections 4–7 in order
  • Paste Combined .gs files from FlockOS-GS/Database/ as Code.gs in each project
  • Set SHEET_ID Script Property in each project
  • Run setup functions: setupAppApi → setupMissionsApi → setupExtraApi → setupExpansion
  • Run initPepper() in John — generates a NEW pepper (old passwords won't work)
  • Seed admin user in AuthUsers + AccessControl tabs
  • Run migrateAllPasswords()
  • Deploy all 4 projects as Web Apps (Execute as Me / Anyone)
  • Update FlockOS-GS/Acts/the_true_vine.js _config with new URLs (Section 8)
  • Test all 4 health checks (Section 9)
  • Clear sessionStorage → test login at the_wall.html
