Purpose

John is the heart of FlockOS. It handles all authenticated operations: member management, role-based access control, pastoral care, worship planning, giving records, event coordination, communications, and more. Every action that requires a login routes through the John API.

With 24 Google Apps Script files and 79 spreadsheet tabs, John is by far the largest and most complex component of the system.

Authentication & Roles

Every John API request must include a valid session token. The 6-tier role system controls what each user can see and do:

RoleLevelAccess
readonly0Personal profile, prayer requests, giving
volunteer1Directory, events, attendance, basic modules
care2Pastoral care cases, assigned member shepherding
leader3Care assignments, statistics, management
pastor4All confidential cases, full override
admin5System configuration, user management, audit

API Pattern

All requests go through doPost() with a JSON body:

{ "action": "care.update", "token": "abc123", ... params }

Api.gs routes the action string to the correct handler function, which validates the role, processes the data, writes to the Sheet, logs the audit trail, and returns JSON.

Key Convention: ID-Based Lookups

All update/delete operations identify records by params.id (a UUID in column A). The backend uses findRowById() from Utilities.gs to locate the row. This is critical — never use row numbers directly, as they shift when rows are added or deleted.

Implementation for Your Church

  1. Create a Google Sheet — this is your main CRM database (the largest one)
  2. Open Apps Script — Extensions → Apps Script
  3. Copy all 24 .gs files into the script editor
  4. Run setupExpansion() — creates all 79 tabs with headers, validation, and default config
  5. Configure Script Properties — set SHEET_ID, SALT, and PEPPER values
  6. Create your admin account — add the first user row directly in the Members tab
  7. Deploy as Web App — set to "Anyone" (the code handles its own authentication)
  8. Test the login flow — verify authentication works before inviting others

Security Notes

  • Passwords are hashed with SHA-256 + salt + pepper — never stored in plain text
  • Session tokens expire after a configurable period
  • Every data modification is logged in the Audit tab with user email, timestamp, and action
  • Confidential pastoral care cases are gated to pastor+ roles at the handler level
  • The requireRole() function guards every handler — no role check, no access
